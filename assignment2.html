1.2 Downward Paths

How should we define a 'location' in an n-ary tree? One natural way to do it is to use a list of integers. (We'll use 0-indexing for it.) No need for a full datatype; we can get away with a type alias, which is just a way of giving some complicated datatype a simple name. (More on that below, once you start your final project.) 

type Path = [Integer]

So empty list means 'root node of the tree', [0] means 'go down to the leftmost node', [1] means 'go down to the one after the leftmost node', and [0,0,1] means 'go down to the leftmost node, then down once more to the leftmost node, then down once more to the next node after the leftmost node'. And so on.

1.2.1 Exercise: treeInsert

In src/Main.hs, define a value called treeInsert with the following type: 

treeInsert :: Tree a -> Path -> Tree a -> Tree a

treeInsert takes a tree to insert, a location in a tree, and a tree to insert the first input into, and returns a new tree with the first tree inserted into the second tree at the specified location.

You should handle bad path inputs in the following way. If the path is so long that it runs downwards off the end of the tree, just have treeInsert return the original second tree input unchanged. Same thing for if the path contains a number so big it would run off the end of the tree rightwards (that is, if the integer at any point in the path exceeds the number of nodes in the tree at that depth level): return the original second tree input unchanged. Finally, if any of the integers in the path are negative, retrun the original second tree input unchanged.

The first step in developing the brains of the game is to think about what software engineers would call our data model. Think of a data model as a high-level description of what types of basic entities your program can work with and how it can manipulate those basic entities. Every program you write is going to have different entities it assumes as basic and different things it'll be able to do to those basic entities. So in general, the data model won't involve low-level stuff like 'integers' and 'addition', unless the program you want to write is a four-function calculator. Rather, it's probably going to involve stuff like like 'user accounts', or 'password questions', or 'students'. And the kinds of manipulations you'll be interested in are going to look less like 'store this bit in a register' and more like 'change this user's password'.

Since this is a game, the kinds of entities you'll be manipulating will be things like a player, an item, a weight, a room, a map, etc. So the next step is to start designing them.

3.1 Interlude: Type Aliases

So one feature of types in Haskell that we haven't gone over yet is type aliases. Type aliases are a nice way to enhance the readability of your code. If you have a big, unwieldy type with lots of parameters, sometimes it helps to give it a nickname you came up with that's going to remind you what it's actually for.

For example, suppose I'm writing a command prompt, similar to the UNIX shell. Maybe I want to give it the feature whereby it remembers anything the user typed in so far, in case the user wants to search back through it rather than retyping it. (Most UNIX shells, such as bash or zsh, already provide this functionality if you type Ctrl-r.)

In that case, maybe I want to have the app store everything the user has typed in so far in a list. Since the user would be typing in strings, a natural way to go here would be to store the history of what the user typed in as a list of strings. In that situation, I could create the following type alias to make it clearer both to myself and other people what this list of strings is actually for: 

type History = [String]

Then later on, if I wanted to write a function that displayed the entire history of the string, I could give it this type: 

displayHistory :: History -> IO ()

Rather than this type: 

displayHistory :: [String] -> IO ()

Don't worry too much about what IO () means for now; we're going to cover that next week. That's just the type we give to a value that performs an action, like printing to the screen.

It's important to remember that a type alias is just there to help you read and keep track of your code. It does not actually define a new datatype. All the above type alias says is: "Hey, Haskell, whenever you see something I'm calling the History type, pretend that instead I wrote [String]." The data keyword is for defining actual new types; the type keyword is just for defining type aliases.

3.3 Interlude: Lookup Tables in Haskell

Every language (pretty much) has some built-in structure for a key-value store. In Python, they're called dictionaries, in Ruby, they're called hashes, and in Javascript, they're called objects.

In Haskell, the most basic form of lookup table is called an association list. An association list is a list of tuples whose first members are keys and whose second members are values. For example, if I wanted to be able to store someone's full name in a table so that I could look it up later based on their username, I could put that information into the following association list: 

names :: [(String, String)]
names = [ ("teichman", "Matt Teichman")
        , ("brady", "Gerry Brady")
        , ("borja", "Borja Sotomayor")
        ]

Association lists come with in-built functions that allow you to look a value up, but that's it. If you want to do things like delete a key-value pair or update the value for a given key, then it's probably not worth hand-rolling these functions yourself. Instead, you can use a data structure called a Map (uppercase 'm', not to be confused with the higher-order function map), which provides a full interface for you to efficiently do key-value storage, lookup, and modification.

(And remember: in Haskell, words like 'modify', 'delete', and 'update' don't mean we actually change the lookup table—instead, these are functions that return a new lookup table with the modification performed, leaving the old one around in memory if we need it.)

The first type variable k in Map k v tells you the type of the lookup table's keys, and the second type variable v tells you the type of the lookup table's values. So for example, a Map String Integer would be a lookup table with strings as keys and integers as values.

You might be wondering why I even mentioned association lists, if their built-in interface is so austere. Well, it turns out that the easiest way to make a Map k v in Haskell is to write out an association list of type [(k,v)] and then use the function fromList in the Data.Map module to convert that association list into a Map k v. This is how I recommend creating Map-s in the next part of the homework.

3.4 Interlude: Qualified Names

We'll be covering Haskell's module system more over the next few weeks. But the thing you need to know in order to start using Map-s is that frequently, different Haskell modules define functions with the same name. When one module has imported all the values of two other modules into its global namespace, and each of those modules contains a function with the same name, this creates a name clash that will make the compiler throw an error.

One option is to just qualify the name of your function with the full module name whenever you use it: 

import Data.Map

λ> :t Prelude.foldr
Prelude.foldr :: Foldable t => (a -> b -> b) -> b -> t a -> b
λ> :t Data.Map.foldr
Data.Map.foldr :: (a -> b -> b) -> b -> Map k a -> b

That's kind of annoying, though. We're going to go with the option of importing Data.Map qualified as M. That will import every value in that module into the global namespace, but prefixing everything M.. Everything in Prelude will be there for you to use unqualified to your heart's content: 

import qualified Data.Map as M

λ> :t foldr
foldr :: Foldable t => (a -> b -> b) -> b -> t a -> b
λ> :t M.foldr
M.foldr :: (a -> b -> b) -> b -> M.Map k a -> b

Ok, back to the adventure game.

We'll encode the exits as pairs consisting of a Direction and a RoomName. The direction represents which direction the exit is in, and the room name represents which room the exit takes you to.

Create a type alias for (Direction, RoomName) called Exit. (This is a bit of a cheat, but we're doing Exit-s this way to make your life a little easier/make the homework take a little less time. More on what makes this a cheat in next week's homework.)

3.7.3 Exercise: Room datatype

Ok, now we're ready to make the Room datatype! Create a record type called Room with four fields: rname of type RoomName, desc of type String, exits of type [Exit], and objects of type [ItemName]. Either derive Show and Eq instances for it or write your own.

When you're done, you should be able to create rooms like this: 

λ> Room "den" "This is where fun comes to live." [(N, "basement")] ["tv", "coffee table"]
Room {rname = "den", desc = "This is where fun comes to live.", exits = [(N,"basement")], objects = ["tv","coffee table"]}

3.8 The Player

Now we're going to make a Player datatype. Every game needs a player, right? The player is going to be you, the user, and it's going to have whatever stats we think a player should have. Just to keep things simple, the player in our game is only going to have three pieces of information that define its current status: an inventory, a number representing the maximum amount of weight you can carry, and a location.

First, create a new file called src/Player.hs and declare the Player module at the top of it: 

module Player where

import Item
import Room

Once again, you have to add Player to the list of exposed library modules in your functional-adventure.cabal project config file: 

library
  hs-source-dirs:      src
  default-language:    Haskell2010
  build-depends:       base
                     , containers
  exposed-modules:     Item
                     , Direction
                     , Room
                     , Player

And then you also have to add it to other-modules in your executable functional-adventure section: 

executable functional-adventure
    hs-source-dirs:      src
    main-is:             Main.hs
    default-language:    Haskell2010
    build-depends:       base
                       , containers
    other-modules:       Item
                       , Direction
                       , Room
                       , Player

I promise you'll get to leave this config file alone eventually. For now we're doing a lot of building, so that means we have a fair amount of editing to do. But once it's set, it'll be set.

3.8.1 Exercise: Player

Create a Player record type with two fields: inventory of type [ItemName], maxWeight of type Integer, and location of type RoomName. Add a deriving clause to the end of it like we've been doing for other datatypes so far: 

-- ...your datatype definition goes here...
deriving (Show, Eq)

When you're done, you should be able to make a Player in the REPL like this: 

λ> Player ["glove", "hat"] 75 "roofdeck"
Player {inventory = ["glove","hat"], maxWeight = 75, location = "roofdeck"}

Player is the first datatype for which we'll need to write a suite of handy utility functions for use in the part of the code that's going to represent the state of the game, and how the state of the game will respond to user input. In general, whenever you create a new datatype, it's good practice to put that datatype into its own module and start writing whatever utility functions it seems like you're going to need. The exact ones you'll need to write from project to project will vary, but one traditional starting point for getting you thinking about this practice is: what getter and setter functions are you going to need?

Luckily, Haskell automatically provides getter functions for us based on the record field names: 

λ> p = Player ["glove", "hat"] 75 "roofdeck"
λ> location p
"roofdeck"
λ> inventory p
["glove","hat"]
λ> maxWeight p
75
λ> :t location
location :: Player -> RoomName
λ> :t inventory
inventory :: Player -> [ItemName]

That will do for many purposes, but it doesn't mean we won't ever want to write a getter function. For example, imagine you had a record type that went 'several levels down'—that is, a record type one of whose fields was a further record type. You might want to write a simple function to help you pull the deeper value out in one step. We won't be bothering with that right now, but once we get to the more complicated record types in this project, it's a good possible need to keep on the lookout for.

3.8.2 Exercise: addItem

Ok, so the first of our setter functions is going to be addItem, which adds an item to a player's inventory. Remember, we're doing everything totally pure! So no mutating. Instead, the function is going to map an ItemName and a Player to a new Player, updated with that ItemName in their inventory: 

addItem :: ItemName -> Player -> Player

Write a function with the above name and type signature that adds an ItemName to a player's inventory. When you're done, you should be able to test it in the REPL like so: 

λ> i = Item "apple" 2
λ> p = Player ["glove", "hat"] 75 "roofdeck"
λ> addItem "apple" p
Player {inventory = ["apple","glove","hat"], maxWeight = 75, location = "roofdeck"}
λ> addItem "pear" . addItem "apple" $ p
Player {inventory = ["pear","apple","glove","hat"], maxWeight = 75, location = "roofdeck"}

3.8.3 Exercise: removeItem

We'll also need a function to remove an item from a player's inventory. (For example, if at some point we write a function for dropping an item, this function will be an important part of that.)

Write a function called removeItem with the following type signature: 

removeItem :: ItemName -> Player -> Player

removeItem is the complement to addItem; it takes an ItemName and a player as input and maps them to a new player with that ItemName removed from their inventory. When you're done writing it, you should be able to test it like this: 

λ> p = Player ["glove", "hat"] 75 "roofdeck"
λ> removeItem "glove" p
Player {inventory = ["hat"], maxWeight = 75, location = "roofdeck"}

There are lots of different ways to write this function and you're welcome to take whatever approach you like. But if you're looking for a simple utility function to help, I recommend taking a look at the Data.List module. Data.List comes with the base package, so you don't have to add anything to your functional-adventure.cabal project config file to be able to use it. All you have to do is import it at the beginning of your Player.hs module, like this: 

module Player where

import Item
import Room

import Data.List

-- ... the rest of your code ...

3.8.4 Exercise: newLocation

Next up is a function to change the location of the player. This will be useful later on for when we write the code that will move the player.

Write a function called newLocation with the following type signature: 

newLocation :: RoomName -> Player -> Player

newLocation will take a room name and a player as input, and return a new player whose location has been updated to be that of the room named. When you're done writing it, you should be able to test it in the REPL like so: 

λ> p = Player ["glove", "hat"] 75 "roofdeck"
λ> p
Player {inventory = ["glove","hat"], maxWeight = 75, location = "roofdeck"}
λ> newLocation "terrace" p
Player {inventory = ["glove","hat"], maxWeight = 75, location = "terrace"}

3.8.5 Exercise: isCarryingAnything

Write a function called isCarryingAnything with the following type: 

isCarryingAnything :: Player -> Bool

isCarryingAnything maps a player to True if that player's inventory has something in it, and False otherwise. When you're done, you should be able to test it in the REPL like this: 

λ> p = Player [] 75 "roofdeck"
λ> p
Player {inventory = [], maxWeight = 75, location = "roofdeck"}
λ> isCarryingAnything p
False
λ> q = addItem "sword" p
λ> q
Player {inventory = ["sword"], maxWeight = 75, location = "roofdeck"}
λ> isCarryingAnything q
True
